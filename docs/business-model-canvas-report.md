# 职业墓场 · 商业模式画布分析报告

> 本文档整理自产品方向讨论与商业模式画布（BMC）梳理，并与当前代码实现对照。  
> 成文日期：2026-04-06。

---

## 1. 产品边界（事实基线）

以下内容以仓库内 `README.md` 与 `src/` 实现为准。

- **形态**：以「数字墓园」方式呈现职业变化的**静态导出站点**（`Next.js App Router + React`）。
- **页面**：首页、归档、职业详情（`/career/[slug]`）、祭奠、关于、404。
- **公开站点基准域名**（canonical / sitemap）：`https://career-graveyard.com`。
- **当前无**：后端、数据库、用户系统、CMS、正式审核流（README 中的后续方向）。

---

## 2. 核心目标（讨论中达成的「脊柱」）

在「权威发布 / 历史展馆 / 职业讨论」等多方向尚未定案的前提下，将**统一目标**表述为两层：

1. **认知**：让用户看到**当下**某职业的真实处境（衰退、冻结、被替代、被重新定价等）。
2. **行动**：若无法或不应再押注该职业，用户能看到**可迁移的方向线索**（相邻职业、技能迁移、学习路径等；可先做到「给线索」再迭代为「给路径」）。

三种产品形态可视为同一脊柱上的不同表现，而非互斥三选一：

| 形态 | 对脊柱的作用 |
|------|----------------|
| 权威 / 信源 | 支撑「当下情况」**可信**（方法论、引用、更新日期）。 |
| 历史展馆 | 用时间线帮助用户**理解变化如何发生**，减轻「只有焦虑没有坐标」。 |
| 讨论 | 用叙事与声音补全数据难以覆盖的体感；若升级为「社区产品」，需单独核算运营成本。 |

## 3. 商业模式画布（九块 + 时间与精力）

### 3.1 客户细分（Customer Segments）

| 细分 | 说明 | 与产品的关系 |
|------|------|----------------|
| A. 泛公众 / 传播者 | 对职业消亡、替代话题有情绪与分享欲 | 流量与传播；**直接付费意愿通常较弱**。 |
| B. 教育与内容从业者 | 教师、课程、科普、自媒体 | 需要可引用结构与案例；**授权 / 合作 / 素材包** 等潜在付费方。 |
| C. 转型中的个体 | 考虑转岗、担心被替代 | 需要信息与行动线索；若加强「路径」与深度内容，**个人付费**潜力更高。 |
| D. 机构（远期） | HR、培训、研究、智库等 | 更可能为**数据、API、报告、定制交付**付费；销售周期更长。 |

**待决策**：主战场是偏 **A+C（公众与转型者）**，还是尽早瞄准 **B 或 D**；这会直接影响渠道与收入设计。

### 3.2 价值主张（Value Propositions）

- **对公众**：在有限浏览内，把抽象的职业变迁变成**可逛、可分享的叙事现场**，而非零散热搜帖。
- **对转型者**（随内容加深）：在单职业语境下看清**衰退逻辑与时间线锚点**，并获得**迁移方向线索**。
- **对 B/D**（若内容与方法论「变硬」）：提供**可引用条目结构**与（未来）可验证来源/更新机制，支撑教学或研究。

当前强项偏 **叙事与策展（体验与情绪）**；**可量化的省钱/省时间** 需在数据产品、工具或课程等形态中强化。

### 3.3 渠道（Channels）

按旅程 **认知 → 考虑 → 购买/深度使用 → 交付 → 售后**：

- **当前较匹配**：社交传播、SEO、设计/独立站曝光；站点即交付（静态托管）。
- **商业化后可能补充**：邮件列表、合作媒体、KOL、企业触达（若做 B2B）；付费则需结账、合同交付与售后通道。

单人或小团队宜侧重 **自有渠道 + 口碑（earned）**；付费投放需在单元经济清晰后再放大。

### 3.4 客户关系（Customer Relationships）

- **当前**：以**自助（Self-service）**为主——用户自行浏览；祭奠投稿见下文「祭奠机制」。
- **若上云 UGC 实时展示**：关系模型趋向**平台责任**，需审核、合规与支持，**运营成本显著上升**。
- **若做 B2B**：常见为**一对一销售 + 项目交付**，与纯静态站阶段差异大。

### 3.5 收入来源（Revenue Streams）

**当前实现无内置付费点**（合理：作品集、文化站或实验阶段）。

可并行探索的方向（均需验证）：

1. **读者向**：赞助、打赏、会员（小额、多笔；依赖价值观契合与持续更新）。
2. **教育向**：教学包、可下载说明、深度报告（依赖版权与来源体系）。
3. **机构向**：API、数据集、定制研究、内训材料（依赖销售与交付能力）。
4. **广告**：技术上可行；与「墓园」气质冲突风险高，**程序化横幅**尤甚；相对可接受的是**标注清晰的栏目赞助**，需单独做品牌与信任权衡。

在未有明确付费产品与访谈验证前，不必硬编 ARPU；应先选定**一条收入假设**再做定价与功能最小集。

### 3.6 关键资源（Key Resources）

- 维护者**时间与技能**（研究、写作、策展、设计、工程、测试）。
- **内容资产**：职业条目结构、叙事模板、（未来）信源与更新节奏。
- **品牌与域名**。
- **工具链**：构建、E2E/视觉测试、托管、（未来）CMS/DB/分析。

商业化情景下，**瓶颈更可能在高质量、可辩护的内容产能**，而非前端技术栈。

### 3.7 关键业务（Key Activities）

- **交付**：新条目、文案与视觉、构建与测试。
- **获客**：运营、SEO、合作拓展。
- **运维**（随扩展增加）：部署、监控、（未来）审核与客户沟通。

README 所列「存储、审核、后台、analytics」若全部落地，**运维与合规**会占用大量周工时，需在时间与精力预算中预留。

### 3.8 关键合作与依赖（Key Partnerships）

- 托管、域名、支付（若收费）、邮件服务等**平台依赖**（需关注条款与迁移成本）。
- **信源与合作方**（若强调可引用）：统计机构、行业报告、学者——可分担背书成本。
- **外包**：设计、法务（尤其 UGC/隐私场景）。

若流量过度依赖单一算法平台，建议用**自有邮件列表 / RSS / 域名**降低风险。

### 3.9 成本结构（Cost Structure）

- **固定**：域名、托管、工具订阅等。
- **变动**：按量 API、广告、外包工时等。
- **一次性**：品牌、首版内容大包等。

静态站阶段**盈亏门槛较低**；商业化后须把**内容生产时间**折算为成本，避免「账面微利、实际透支」。

### 3.10 时间与精力预算（单人/小团队必加项）

将 3.7 中活动拆解为**每周可承受小时数**，总和须低于可持续上限（常需为休息与学习留白）。

- 若祭奠改为**云上实时展示 + 审核**：固定增加人工环节，不可自动化部分决定能否 solo。
- 若承接 **B2B 定制**：与「可扩展产品」共享同一时间池，需明确优先级。

**规则**：时间账不平则商业模式在「人」这一层不成立，应在扩张前调整范围或外包/自动化。

## 4. 画布一致性自检（摘要）

| 检验项 | 要点 |
|--------|------|
| 价值 ↔ 细分 | 公众要共鸣与可分享性；转型者要结构与下一步；勿用 B2B 话术覆盖全体。 |
| 收入 ↔ 价值 | 缺少可重复交付的「硬价值」前，订阅或高价难以成立。 |
| 渠道 ↔ 细分 | 若主打机构，渠道应对齐 LinkedIn、行业活动、内推等，而非仅泛娱乐流量。 |
| 活动 ↔ 时间 | 扩展越多，越需先砍清一条主路径。 |
| 成本 ↔ 收入 | 先锁定一条收入假设，再反推最低内容与功能集。 |

## 5. 单元经济（何时细算）

在出现**真实付费**后，建议粗算：

- **CAC**：获客相关支出（含时间摊销）÷ 新客户数。
- **LTV**：ARPU × 平均生命周期（月）。
- **回本周期**：CAC ÷ 月 ARPU。

经验方向：LTV 应显著高于 CAC；回本周期不宜过长。数字不达标时，优先动**定价、渠道、留存**，而非只堆功能。

## 6. 祭奠机制（与实现对齐）

讨论中已澄清：**祭奠并非「用户浏览器 localStorage 持久化悼词」的单一模式**（若 README 仍写旧表述，建议修订为与代码一致）。

**当前实现要点**（`src/app/layout.js`、`src/components/Shell.js`、`src/data.js`）：

- 页面上展示的悼词列表来自 **`initialMemorials` 等静态数据**，属策展/示例；文案说明其为**静态示例，非访客实时投稿**。
- 访客通过表单生成 **mailto 邮件草稿**（或按备用说明手动发信）至**项目邮箱**；**不会自动上墙**；维护者以邮箱为**素材池**，再决定是否通过**更新静态站点**纳入公开内容。
- `localStorage` 在站内的用途包括**语言/区域偏好**等，**不是**祭奠正文的云端存储方案。

**结论**：不是「论坛式云端实时留言板」，也不是「悼词仅存本机」；而是 **邮箱异步投稿 + 人工筛选 + 静态发布**。若未来改为实时云上留言，须同步规划**审核、法务与滥用**成本。

## 7. 机构付费与广告（结论摘要）

- **机构付费**：可能，但通常购买的是**可信度、结构化交付、数据或研究价值**，而非单纯氛围；需方法论、授权边界与更新机制。
- **广告**：可做；**程序化贴片**与调性冲突风险高；**透明赞助/栏目合作**相对可控，需单独决策。

## 8. 待你方拍板的开放问题

1. **长期定位**：偏公共艺术/文化站，还是可商业化的职业情报与学习产品？
2. **付费方优先级**：更愿服务**读者（小额多次）**还是**机构（大额少次）**？
3. **祭奠产品形态**：维持「邮箱 + 静态策展」，还是演进为「云上 UGC」（将显著改变关系模型与成本）？
4. **多方向取舍**：在「权威 / 展馆 / 讨论」中，**第一版必须兑现**用户承诺的是哪一条（建议与第 2 节脊柱对齐）？
