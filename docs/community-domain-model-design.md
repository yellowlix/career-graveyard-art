# 职业墓场 V1（邮箱投稿版）领域与技术设计

## 1. 文档目标

本文档用于固定 `career-graveyard-art` 的 `V1` 产品范围、核心流程和技术实现方向。

当前 `V1` 的定位是一个静态职业档案站，而不是一个带后台和数据库的社区平台：

- 公开站点继续提供职业档案展示
- `Memorial` 页保留留言表单，但提交走 `mailto:`
- `About` 和 `Memorial` 页都提供项目邮箱
- 新增职业和祭奠留言都通过同一个项目邮箱处理
- 站点不接入后端数据库，也不承担真实用户投稿的在线存储职责

本文档只定义当前 `V1` 的落地范围；如未来要升级为数据库版，请参考附录中的后续草案。

## 2. V1 产品边界

### 2.1 V1 要做什么

- 展示已收录职业档案
- 展示职业详情页
- 保留 `Memorial` 表单，允许用户填写职业、署名和悼词内容
- 通过 `mailto:` 生成邮件草稿，把祭奠留言发送到项目邮箱
- 在 `Memorial` 和 `About` 页提供新增职业的邮箱联系入口
- 在 `Memorial` 页保留静态示例悼词，用于展示页面氛围

### 2.2 V1 不做什么

- 不做后端数据库
- 不做站内新职业提名流程
- 不做留言审核后台
- 不做管理员登录与管理页
- 不做留言举报
- 不做复杂权限系统
- 不做多语言建模
- 不做证据来源管理

## 3. 关键产品判断

### 3.1 社区能力定义

`V1` 的社区能力是“邮箱投稿能力”，不是“在线社区共创能力”。

也就是说：

- 用户可以通过页面生成投稿邮件
- 用户不能在站内直接发布真实留言
- 用户不能在站内修改职业档案
- 职业档案最终仍由维护者手工整理和发布

### 3.2 新增职业方式

新增职业仍通过项目邮箱联系，不在站内完成。  
这是一条运营入口，不是站内业务流程。

### 3.3 数据来源判断

`V1` 的正式展示数据继续来自本地静态文件：

- 职业档案来自 `src/data.js`
- 页面文案和邮箱配置来自 `src/data.js`
- 静态示例悼词来自 `src/data.js`

用户原始投稿不写入站点，不写入数据库，而是进入项目邮箱收件箱。

## 4. 技术栈固定方案

### 4.1 V1 技术栈

- 前端构建：`Vite`
- 页面结构：原生多页面 `HTML`
- 前端逻辑：原生 `JavaScript (ES Modules)`
- 样式：原生 `CSS`
- 自动化测试：`Playwright`

### 4.2 为什么当前不引入后端

当前产品复杂度还不要求引入数据库或后台：

- 页面数有限
- 展示逻辑简单
- 投稿通过邮箱即可完成最小闭环
- 真实内容整理仍以人工策展为主

因此 `V1` 优先做“静态展示 + 邮箱投稿”，而不是“在线数据平台”。

## 5. 页面与能力映射

### 5.1 首页

- 读取本地静态职业档案精选集
- 继续承担品牌表达和入口作用

### 5.2 Archive 页

- 展示正式职业档案列表
- 按状态筛选
- 按名称或年份排序

### 5.3 职业详情页

- 按 `slug` 读取本地静态职业档案
- 展示时间线、证言、因素
- 展示规则计算得到的相关职业

### 5.4 Memorial 页

- 作为单页双模式邮箱投稿页存在
- 模式一：`祭奠已收录职业`
- 模式二：`新增职业投稿`
- 两个模式都通过 `mailto:` 生成邮件草稿并发送到同一个项目邮箱
- 页面展示静态策展示例，但不展示真实用户实时投稿
- 不接入数据库，不提供在线发布、审核后台或实时社区流

### 5.5 About 页

- 展示项目说明
- 提供项目邮箱入口
- 保留 GitHub 协作入口
- 明确区分“祭奠留言 / 新增职业”与“代码协作 / Issue / PR”

## 6. 数据组织方式

### 6.1 正式展示数据

站点正式展示数据保存在本地静态文件中，当前以 `src/data.js` 为主：

- 职业主数据
- 时间线
- 因素
- 证言
- 页面文案
- 联系邮箱配置
- 静态示例悼词

### 6.2 用户原始投稿

用户原始投稿不保存在站内，而是进入项目邮箱收件箱：

- 祭奠留言邮件
- 新增职业投稿邮件

网站本身不承担真实留言存储职责，也无法自动公开、统计或审核投稿。

### 6.3 维护者工作流

当前版本的内容更新链路为：

1. 用户通过页面生成邮件草稿
2. 用户使用本地邮件客户端发送到项目邮箱
3. 维护者在邮箱中接收和筛选投稿
4. 维护者手工整理后更新静态数据
5. 重新部署站点

### 6.4 V1 静态数据结构规范

当前 `V1` 不引入数据库，因此实现时应把页面依赖的结构化内容收敛在静态配置层，避免在 `app.js` 中临时拼装。

推荐最小结构如下：

- `siteMeta.contactEmail`
  - 单一项目邮箱配置源
- `siteCopy.memorial`
  - 双模式切换文案
  - 各模式独立的字段标签、占位符、说明、校验提示、按钮文案
- `siteCopy.memorialEmail`
  - 已收录职业祭奠邮件模板
- `siteCopy.contactEmail`
  - 新职业投稿邮件模板
- `initialMemorials`
  - 按模式拆分的静态示例内容
  - 仅用于展示，不与真实用户投稿混用

推荐静态示例结构保持为：

- `existing[]`
  - `careerSlug / signature / date / text`
- `unlisted[]`
  - `careerName / introduction / signature / date / text`
  - `references` 为可选字段

推荐祭奠邮件模板配置至少包含：

- `subjectPrefix`
- `bodyTemplate`
- `emptySignature`
- `emptyText`

推荐新增职业邮件模板配置至少包含：

- `subjectPrefix`
- `bodyTemplate`
- `referencesLabel`
- `emptyIntroduction`
- `manualCopyHint`

实现约束：

- 联系邮箱只能从一个配置字段读取，不在模板和脚本中重复写死
- 示例内容与真实投稿渠道必须分层，不能在运行时混合
- 页面使用的邮件文案必须可配置，不能把主题前缀硬编码在事件处理函数中

## 7. 邮件规范

### 7.1 联系邮箱

当前文档阶段统一使用项目邮箱占位值：

- `contact@career-graveyard.art`

实现前应替换为真实项目邮箱，但页面和配置层都应只从单一配置源读取。

### 7.2 祭奠留言邮件

邮件主题格式固定为：

- `[Career Graveyard Memorial] {职业名} - {署名}`

邮件正文模板固定包含：

- `职业：{职业名}`
- `署名：{署名}`
- `悼词：{正文}`
- `来源页面：Memorial`

### 7.3 新增职业邮件

邮件主题格式固定为：

- `[Career Graveyard Submission] {职业名} - {署名}`

邮件正文固定包含：

- 职业名称
- 职业简介
- 悼词
- 署名
- 可选参考资料
- 来源页面：`Memorial`

### 7.4 `mailto:` 的产品边界

`mailto:` 只能帮助用户生成邮件草稿：

- 网站无法确认用户是否真正发出邮件
- 网站无法自动公开投稿
- 网站无法自动统计或审核投稿

页面应提供 fallback 文案：如果邮件客户端未自动打开，用户可以手动复制邮箱并按页面模板发送。

### 7.5 `Memorial` 页交互与 fallback 验收规则

以下规则优先于本节旧描述执行：

- `Memorial` 页为单页双模式邮箱投稿页，默认模式为 `祭奠已收录职业`
- 切换项采用轻量文字标签 + 激活下划线，不做后台化分段按钮
- 两个模式的输入状态独立保留，切换后不互相覆盖
- 切换区与正文区共用同一条左右内缩边界
- `祭奠已收录职业` 包含职业选择、署名、悼词内容
- `新增职业投稿` 包含职业名称、职业简介、署名、悼词内容，参考资料为选填
- 两个模式都在必填项未完成前保持禁用，邮件主题与正文预览始终实时更新
- fallback 区至少展示邮箱、主题预览、正文预览，确保 `mailto:` 未自动唤起时仍可手动发送
- 示例区必须按模式切换，并明确标注为静态示例，不得误导为实时投稿流

为避免实现阶段出现不同理解，当前 `V1` 的交互规则固定如下：

#### 表单输入

`Memorial` 页保留三个输入：

- 职业
- 署名
- 悼词内容

职业字段只允许选择已收录职业，不提供“其他职业”选项。

#### 提交动作

- 点击提交后，前端立即尝试生成 `mailto:` 跳转
- 邮件收件人为项目邮箱
- 邮件主题和正文使用本文件定义的固定模板
- 网站不显示“已提交成功”这类服务端确认型文案

#### 页面反馈

提交后页面应提供可见的 fallback 区域或静态说明，至少满足以下其一：

- 明确展示项目邮箱，便于手动复制
- 明确展示邮件主题格式
- 明确提示用户可将页面中的表单内容手动复制到邮件中发送

当前推荐做法是三者都提供，以降低 `mailto:` 在不同设备和浏览器下的不确定性。

#### 示例悼词区

示例悼词区必须明确标注为“静态示例”或“策展示例”，不能继续使用容易被理解为实时投稿流的标题。

推荐标题方向：

- `Curated Tributes / 示例悼词`
- `Sample Memorials / 示意悼词`

页面上应至少有一句说明，明确这些内容不是当前访客实时提交的结果。

#### 不允许出现的误导性表述

当前 `V1` 页面中不应再出现：

- “最近悼词”如果它会让用户误解为实时流
- “提交后会公开展示”
- “留言进入待审”
- “留言保存在本地浏览器”
- 任何暗示网站能确认邮件已成功发送的文案

## 9. 实施路线

### Phase 1：文档收口

- 更新主文档到邮箱投稿版口径
- 将数据模型附录降级为未来草案
- 固定邮件主题、正文模板和邮箱占位值

### Phase 2：前端行为调整

- `Memorial` 页移除本地存储和动态公开逻辑
- 表单改为生成 `mailto:`
- 保留静态示例悼词
- `About` 与 `Memorial` 页统一邮箱文案

### Phase 3：测试与回归

- 保持现有首页、Archive、详情页回归测试
- 新增或更新 `mailto:`、静态示例悼词和邮箱文案相关测试

## 10. 成功标准

满足以下条件即可认为当前 `V1` 达成：

- `Memorial` 表单可生成正确的 `mailto:` 邮件
- `Memorial` 页提供明确的邮箱 fallback 信息，用户即使未自动唤起邮件客户端也知道如何继续投稿
- 页面中不再承诺本地存储、待审核或公开同步
- 用户能明确知道新增职业和祭奠留言都通过项目邮箱处理
- `Memorial` 页保留静态示例悼词，但不会误导为真实实时投稿
- 站点继续以本地静态数据作为正式展示源

## 11. 非目标

本版不覆盖：

- 数据库
- 管理页
- 管理员登录
- 留言待审流
- 在线真实留言列表
- 候选职业
- 站内职业提名
- 复杂权限系统
- 多语言表设计
- 证据来源表

如后续范围扩大，应在未来版本中处理，而不是继续向当前 `V1` 叠加复杂性。

## 12. 决策摘要

- 运行时数据主来源：本地静态文件
- 前端技术栈：继续保留 `Vite + HTML + JS + CSS`
- 后端数据库：不使用
- 管理页：不实现
- 留言提交：走项目邮箱
- 新增职业：走项目邮箱
- 示例悼词：静态展示，不代表实时用户投稿
